# OpenClaw Life Coach 路由策略

## 1. 文档目标

这份文档定义 OpenClaw Life Coach 在一次会话中如何决定：
- 当前输入是否应先进入 safety
- 当前最适合调用哪个 skill
- 是否需要读取 knowledge
- 是否需要参考 memory
- 是否要走多模态处理链路

目标不是让系统“尽量多调用能力”，而是让系统在合适的时候只调用最必要的能力。

---

## 2. 路由总原则

### 2.1 安全优先
任何时候，只要出现明显风险信号，优先进入 safety routing，而不是继续普通 coaching。

### 2.2 少而准
一次响应尽量只选择一个主 skill。
不要把 goal-clarify、emotional-debrief、weekly-review 混在一次回复里同时展开。

### 2.3 先判断场景，再决定能力
先识别用户当前是在：
- 倾诉情绪
- 澄清目标
- 拆执行阻碍
- 做周期复盘
- 提供多模态材料

再决定是否需要：
- skill
- knowledge
- memory
- multimodal adapter

### 2.4 优先最小有效路径
如果普通教练式回应就足够，不强行调用复杂链路。
如果一个 skill 足够，不额外叠加多个 skill。
如果 knowledge 不会显著提升质量，就不注入。

---

## 3. 会话级路由顺序

建议按以下顺序执行：

1. 接收用户输入
2. 判断输入模态
3. 做 safety 初筛
4. 判断是否需要读取 memory
5. 识别当前会话场景
6. 选择主 skill 或 default-coach-response
7. 判断是否需要 knowledge 注入
8. 组合 prompts + skill + memory + knowledge
9. 生成回复
10. 输出记忆更新候选

这条顺序的核心是：

> 先决定能不能继续，再决定怎么继续。

---

## 4. Safety 优先级规则

### 4.1 必须优先进入 safety 的场景
出现以下任一情况，普通 coaching 暂停：
- 明确自伤、自杀、伤害他人表达
- 明显危机或现实安全威胁
- 请求医疗、诊断、治疗判断
- 请求法律、投资等高责任确定性建议
- 强烈依赖性表达，要求 agent 替代现实支持系统

### 4.2 可以继续 coaching 但需收紧表达的场景
以下情况可以继续，但需要进入 `medium` 风险模式：
- 明显情绪激烈波动
- 强烈自责、羞耻、绝望表达
- 长期反复提到无价值感、极端化判断
- 用户对 agent 产生明显投射或过度理想化

此时要求：
- 降低解释强度
- 降低行动要求
- 不做强推进
- 优先稳定、澄清和现实支持建议

### 4.3 可以继续普通路由的场景
- 一般性情绪倾诉
- 目标模糊
- 执行受阻
- 周复盘
- 习惯中断
- 上传图片、语音、文件用于普通教练辅助

---

## 5. 主 skill 路由规则

## 5.1 `coach-intake`
### 触发条件
- 新用户首次进入系统
- 用户明确希望建立长期陪伴关系
- 当前需要补齐长期目标、偏好、边界等基础画像

### 信号示例
- “你先了解一下我吧”
- “我想建立一个长期教练关系”
- “你可以先认识一下我的情况”

### 不适合触发的情况
- 用户已经在讨论一个非常具体的问题
- 用户处于明显情绪事件中，需要先承接

---

## 5.2 `goal-clarify`
### 触发条件
- 用户想改变，但目标模糊
- 用户方向很多，无法聚焦
- 用户停留在愿望层，没有进入行动定义

### 信号示例
- “我想把生活调整好，但不知道从哪开始”
- “我有很多想做的事，但很乱”
- “我其实不知道自己真正想推进什么”

### 常见排除条件
- 用户已经有清晰目标，只是执行卡住
- 用户当前主要在处理一次情绪事件

---

## 5.3 `emotional-debrief`
### 触发条件
- 用户刚经历一次强情绪事件
- 用户反复描述某次冲突、挫败、误解、羞耻体验
- 用户需要从混乱感受走向理解与整理

### 信号示例
- “刚才和同事吵完，我现在很乱”
- “我今天被一句话刺到了，一直缓不过来”
- “我知道自己反应过度了，但我不知道为什么”

### 常见排除条件
- 用户要做的是一周整体复盘
- 用户主要是目标不清，而不是事件冲击

---

## 5.4 `weekly-review`
### 触发条件
- 用户明确在做周复盘
- 用户希望总结一个阶段的进展、阻碍与下周行动
- 用户在混乱节奏后希望重新聚焦

### 信号示例
- “帮我做个这周复盘”
- “我想总结一下这周到底发生了什么”
- “我感觉这周很乱，想重新理清节奏”

### 常见排除条件
- 用户只是在谈单次事件
- 用户尚未形成任何阶段性信息，先要做 intake 或 clarify

---

## 5.5 `habit-reset`
### 触发条件
- 用户提到习惯中断、计划掉线、节奏崩掉
- 用户重点不是理解深层目标，而是尽快恢复执行

### 信号示例
- “我又断掉了三天”
- “计划一停就回不去了”
- “我现在想先恢复节奏”

### 常见排除条件
- 用户真正卡在目标定义，而不是习惯恢复
- 用户情绪强度过高，需要先 debrief

---

## 5.6 `voice-checkin`
### 触发条件
- 输入模态为语音
- 用户明确要做语音倾诉、语音 check-in

### 调用前提
- 先完成音频转写
- 再根据转写文本决定是否继续分流到其他 skill

### 路由原则
`voice-checkin` 可以既是入口 skill，也可以是多模态预处理链路。
如果转写后发现本质是周复盘，则后续主 skill 可切到 `weekly-review`。

---

## 5.7 `vision-reflection`
### 触发条件
- 输入模态为图片
- 用户上传聊天截图、计划截图、环境图片，希望做情境分析或反思

### 调用前提
- 先完成图像摘要
- 再根据摘要判断是普通观察、情绪事件还是执行问题

### 路由原则
`vision-reflection` 通常作为图像入口 skill。
如果图像内容只是辅助材料，真正主任务仍可切换到别的 skill。

---

## 5.8 `default-coach-response`
### 触发条件
- 用户表达不够结构化，但也不明显命中某个 skill
- 当前只需要一个简短澄清问题来继续
- 不值得立刻进入完整 skill 流程

### 适用示例
- “我最近状态不太对。”
- “我不知道该怎么说。”
- “就是感觉哪里都不顺。”

默认响应目标：
- 先承接
- 再聚焦
- 为下一轮判断创造足够信息

---

## 6. 路由冲突处理规则

### 6.1 情绪事件 vs 目标澄清
如果用户同时表达“我很乱”和“我想改变”，优先判断：
- 若当前被具体事件强烈占据，先 `emotional-debrief`
- 若当前更像长期方向混乱，先 `goal-clarify`

### 6.2 周复盘 vs 习惯重启
如果用户在复盘中提到节奏中断：
- 如果重点是总结全周，主 skill 为 `weekly-review`
- 如果重点是立刻恢复行动，主 skill 为 `habit-reset`

### 6.3 多模态入口 vs 真实任务
如果用户上传了语音/图片，不能默认多模态 skill 就是最终主 skill。
应区分：
- 模态入口 skill：负责把材料转成结构化文本
- 主任务 skill：负责真正的教练流程

---

## 7. Memory 调用规则

### 7.1 应优先读取 memory 的场景
- 用户在谈长期目标
- 用户在重复出现的卡点上再次求助
- 用户沟通风格和明确边界会影响本次回应
- 用户正在做周期复盘，需要对照长期主题

### 7.2 可以不读取 memory 的场景
- 一次性短问答
- 非持续关系场景
- 当前问题足够局部，memory 不会提高回答质量

### 7.3 调用原则
- 只读取和当前问题相关的记忆
- 优先读长期目标、稳定偏好、明确边界、有效支持方式
- 不把一次性情绪碎片当成优先上下文

---

## 8. Knowledge 调用规则

### 8.1 应调用 knowledge 的场景
- 当前 skill 明确，需要稳定方法框架支撑
- 用户问题涉及反复出现的结构化主题
- 当前需要更稳定的提问路径或判断框架
- safety 场景需要调用 safety 知识块

### 8.2 不应调用 knowledge 的场景
- 单纯承接性回应即可完成目标
- 调用知识只会让回答变重、变硬、变像讲课
- 当前需要先澄清信息，尚未到框架介入阶段

### 8.3 调用原则
- 优先少量高相关知识块
- 优先摘要，不整篇灌入
- 优先与当前 skill 匹配的知识块
- safety 场景优先 safety 类知识

---

## 9. 多模态分流规则

### 9.1 文本输入
默认直接进入 safety + skill 路由。

### 9.2 语音输入
先进入：
- `audio -> transcript`
- 提取主题、情绪强度、请求类型
- 再进入 safety + skill 路由

### 9.3 图片输入
先进入：
- `image -> scene summary`
- 提取场景、文本内容、关系线索、任务线索
- 再进入 safety + skill 路由

### 9.4 文件输入
先进入：
- `file -> structured summary`
- 提取文档主题、结构、待分析对象
- 再进入 safety + skill 路由

### 9.5 原则
多模态能力的任务不是直接生成最终回答，
而是把非文本材料规范化为主 agent 可消费的结构化输入。

---

## 10. 首版可实现的规则路由结构

建议首版输出统一结构：

```json
{
  "modality": "text|audio|image|file",
  "safety": {
    "riskLevel": "low|medium|high",
    "allowCoaching": true,
    "needsSafetyMode": false
  },
  "route": {
    "primarySkill": "goal-clarify",
    "fallback": "default-coach-response",
    "needsKnowledge": true,
    "needsMemory": true,
    "needsMultimodalAdapter": false
  }
}
```

这样后续无论是规则引擎还是 plugin，都有统一的执行出口。

## 12. 最终结论

路由策略的核心不是“让系统更复杂”，而是：

> 在对的时机，调用对的能力，并且尽量少调、精调、稳调。

对 OpenClaw Life Coach 来说，正确顺序始终应该是：

> safety 优先 → skill 聚焦 → knowledge/memory 精准补充 → 多模态作为输入增强

只有这样，系统才不会变成“什么都懂一点，但每次都乱用”的拼装体。
