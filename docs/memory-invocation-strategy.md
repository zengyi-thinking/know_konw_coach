# OpenClaw Life Coach Memory Pipeline 设计

## 1. 文档目标

这份文档定义 OpenClaw Life Coach 如何处理记忆。

目标不是“记住更多聊天内容”，而是建立一套可控的长期记忆机制，使系统能够：
- 逐步理解用户
- 避免记忆污染
- 支持长期陪伴感
- 随时间归档、降级、清理无关信息

---

## 2. 记忆系统的核心原则

### 2.1 宁少勿乱
长期记忆的质量比数量重要。
如果把大量一次性碎片写进 memory，系统只会越来越混乱。

### 2.2 只记长期有价值的信息
memory 应优先记录：
- 长期目标
- 稳定偏好
- 明确边界
- 已验证有效的支持方式
- 反复出现的长期阻碍模式

### 2.3 不把瞬时情绪当长期画像
用户一次沮丧、一次抱怨、一次低落，不应直接写成长期人格标签。

### 2.4 新事实优先于旧推测
如果新对话明确推翻旧记忆，应优先更新或降级旧记忆。

### 2.5 允许归档与删除
memory 不是只增不减。
长期系统必须具备归档、降级、删除和自净化能力。

---

## 3. 记忆分层结构

建议分成三层：

### 3.1 short-term session memory
只服务当前会话。
包括：
- 当前主题
- 当前情绪状态
- 本轮已确认的上下文
- 本轮暂时性结论

特点：
- 生命周期短
- 不直接进入长期库
- 会话结束后可丢弃或仅作为候选来源

### 3.2 long-term user memory
长期有效的用户信息。
包括：
- 长期目标
- 稳定偏好
- 沟通风格偏好
- 明确边界
- 有效支持方式
- 重复模式

特点：
- 结构化保存
- 需通过评分和校验后才写入
- 应能被后续会话读取与更新

### 3.3 archived memory
过期但保留追溯价值的信息。
包括：
- 已不再成立的旧目标
- 早期但已失效的偏好
- 曾经出现但后来被修正的模式

特点：
- 默认不进入普通上下文
- 保留历史轨迹
- 用于冲突追溯或长期演化分析

---

## 4. 应保留的长期记忆类型

### 4.1 长期目标
例如：
- 想建立稳定作息
- 想改善工作节奏
- 想减少关系中的过度自责

### 4.2 稳定偏好
例如：
- 喜欢简洁直接表达
- 不喜欢太鸡汤
- 更接受行动导向建议

### 4.3 明确边界
例如：
- 不希望深挖某类家庭话题
- 不喜欢高压式推进
- 不想接受过度情绪安抚风格

### 4.4 已验证有效的支持方式
例如：
- 先澄清再给动作对用户更有效
- 最小行动比完整计划更容易执行
- 周复盘比日复盘更适合该用户

### 4.5 重复出现的长期阻碍模式
例如：
- 目标过多导致分散
- 情绪波动后执行中断
- 一遇到评价就过度自责

---

## 5. 不应进入长期记忆的内容

以下内容默认不写入长期 memory：
- 一次性情绪爆发
- 临时抱怨
- 当天偶发状态
- 无法验证的随口表达
- 与长期成长无关的细小事实
- 非必要高敏感隐私

原则：

> 如果这条信息在未来对教练质量没有明确帮助，就不要记。

---

## 6. Memory Pipeline 核心模块

建议拆成四个子模块：

- `collector`
- `scorer`
- `merger`
- `archive`

---

## 6.1 collector
### 目标
从对话中提取记忆候选，而不是直接落库。

### 输入
- 当前对话内容
- 当前 skill
- 会话摘要
- 本轮用户明确表达的新事实

### 输出
- 记忆候选列表
- 候选类型
- 原始证据片段

### 提取重点
优先提取：
- 明确长期目标
- 稳定偏好
- 边界表达
- 被重复确认的长期模式
- 已验证有效的支持方式

### 不应直接提取为候选的内容
- 纯情绪宣泄
- 一次性事件细节
- 未形成结论的猜测

---

## 6.2 scorer
### 目标
给每条候选打分，决定是否值得进入长期记忆。

### 推荐评分维度
- `valueScore`：长期价值
- `confidenceScore`：信息可信度
- `stabilityScore`：是否可能长期稳定
- `sensitivityScore`：是否过于敏感

### 典型判断思路
#### 高分候选
- 用户明确、多次表达的长期目标
- 用户明确指出的沟通偏好
- 已多次验证有效的支持方式

#### 低分候选
- 只出现一次的情绪判断
- 模糊、暧昧、未确认的信息
- 高敏感但对教练无必要的信息

### 落库建议
- 高价值 + 高置信：可进入长期库
- 中价值 + 中置信：保留为观察候选
- 低价值或高敏感：丢弃

---

## 6.3 merger
### 目标
把新候选与已有长期记忆进行合并、更新或冲突检测。

### 处理类型
- `new`：新增
- `merge`：补充已有记忆
- `upgrade`：由弱结论升级为强结论
- `downgrade`：旧结论被削弱
- `conflict`：与旧记忆明显冲突
- `discard`：不进入长期库

### 合并原则
- 相同主题优先合并，不重复堆积
- 新证据更强时，可升级旧记忆
- 新证据与旧记忆冲突时，不直接覆盖，先标冲突并降级旧记忆

### 示例
旧记忆：
- 用户偏好长篇解释

新证据：
- 用户明确说“简短直接更适合我”

处理：
- 旧记忆降级或归档
- 新记忆进入长期库

---

## 6.4 archive
### 目标
处理过期、冲突、低价值和冗余记忆，保持长期库干净。

### 应归档的情况
- 已不再成立的旧目标
- 过期偏好
- 被新事实取代的旧结论
- 长期未再出现的低价值主题

### 应删除的情况
- 重复冗余
- 明显错误
- 非必要敏感内容
- 后续证据证明无效的推测

### 应降级为观察态的情况
- 只有一次证据
- 还未被验证的偏好
- 可疑但暂时不能完全删除的信息

---

## 7. 记忆写入规则

长期记忆写入应满足至少以下条件中的大部分：
- 对未来教练质量有明确帮助
- 具有一定长期稳定性
- 用户表达足够明确
- 有重复证据或较高置信度
- 不属于非必要敏感隐私

建议首版采用规则：

### 可写入
- 用户明确陈述的长期目标
- 用户明确陈述的稳定偏好
- 用户明确设置的边界
- 已多次验证有效的支持方式
- 重复出现的长期阻碍模式

### 暂不写入
- 一次性情绪判断
- 仅单轮出现且价值不高的信息
- 可能引发误画像的推测

---

## 8. 记忆读取规则

不是每次都要读全部 memory。

### 8.1 应读取的场景
- 用户在讨论长期目标
- 用户再次回到已知长期卡点
- 用户边界和偏好会明显影响回应方式
- 周期复盘需要对照长期主题

### 8.2 可以不读取的场景
- 一次性短问答
- 当前问题足够局部
- 当前更需要先澄清信息，memory 不会提升质量

### 8.3 读取原则
- 只取相关记忆
- 优先长期目标、稳定偏好、明确边界、有效支持方式
- 避免把大量旧碎片放进上下文

---

## 9. 冲突记忆处理规则

当新旧记忆冲突时，不应简单覆盖。

建议策略：
1. 保留旧记忆的历史记录
2. 将旧记忆降级或归档
3. 将新记忆标记为当前有效版本
4. 在后续对话中继续观察验证

例如：
- 旧记忆：用户喜欢详细分析
- 新记忆：用户最近明确表示希望更短、更直接

处理方式：
- 新记忆成为当前有效偏好
- 旧记忆归档为历史版本

---

## 10. 自净化规则

自净化的目标不是“删得越多越好”，而是让记忆保持高信噪比。

### 10.1 去无关
移除与长期陪伴无关的碎片信息。

### 10.2 去污染
移除偶发、误判、未经验证的信息。

### 10.3 去冗余
合并重复内容，避免多条表达同一件事。

### 10.4 去漂移
防止 agent 把偶发情绪或一次性事件误当成长期人格画像。

---

## 11. 首版可实现的数据结构

建议首版用结构化字段：

```json
{
  "id": "memory-preference-direct-style",
  "type": "preference",
  "content": "用户更喜欢简洁直接的表达方式",
  "confidence": 0.88,
  "status": "active",
  "evidenceCount": 2,
  "source": ["session-2026-04-05"],
  "tags": ["style", "communication"],
  "updatedAt": "2026-04-05"
}
```

状态建议：
- `active`
- `observing`
- `archived`
- `discarded`

---

## 12. 首版执行流程建议

一次会话结束后：

1. `collector` 抽取候选
2. `scorer` 打分
3. `merger` 与现有长期记忆对比
4. `archive` 处理低价值、冲突、过期信息
5. 输出：
   - 新增记忆
   - 更新记忆
   - 归档记忆
   - 删除建议

推荐输出结构：

```json
{
  "new": [],
  "updated": [],
  "archived": [],
  "discarded": []
}
```

---

## 13. 与 skill / knowledge 的联动

### 13.1 skill 对 memory 的影响
不同 skill 更容易产出不同类型记忆：
- `coach-intake`：目标、边界、偏好
- `goal-clarify`：长期目标、关键阻碍
- `weekly-review`：重复模式、有效策略
- `emotional-debrief`：可重复出现的触发模式（仅在多次验证后）

### 13.2 memory 对 knowledge 的影响
如果 memory 显示用户长期反复卡在某类主题，可提高对应知识块权重。

但原则仍然是：

> 当前输入优先，memory 提供辅助，不主导一切。

---

## 14. 实现阶段建议

### P0
先写清楚：
- 记什么
- 不记什么
- 如何评分
- 如何合并
- 如何归档

### P1
再补：
- 结构化模板
- 状态字段
- 规则引擎

### P2
后续增强：
- LLM 提取候选
- 冲突评分
- 过期衰减
- 自动归档建议

---

## 15. 最终结论

Memory Pipeline 的核心，不是“让 agent 记住更多”，而是：

> 让它记住真正值得记住的东西，并持续忘掉不值得记住的东西。

对 OpenClaw Life Coach 来说，理想的长期记忆应该是：
- 少量
- 准确
- 可更新
- 可归档
- 可验证

只有这样，系统才会越来越懂用户，而不是越来越被旧噪声拖累。
