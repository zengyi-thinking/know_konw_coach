# 产品设计问答整理

## 目标

本文用于整理本轮围绕 `Life Coach`、`OpenClaw`、`skill`、`agent`、`调料包`、`自反馈学习` 与跨产品迁移所展开的产品设计问答，作为后续继续收敛产品定义与架构边界的讨论底稿。

## 讨论范围

本轮问答主要覆盖以下主题：

- 如果以 `Life Coach` 模式长期协作，系统应如何在时间线上陪伴用户
- 一个 agent 如何理解自己的成长与进化
- 一个 `Life Coach` 调料包需要具备哪些关键元素，才能“越用越对味”
- 是否适合使用多个 agent 分别管理记忆、人格与行为边界
- 在 `OpenClaw` 中，调料包是否可以定义自己的 agent 工作流
- 如果 `OpenClaw` 本身已经具备自我演化能力，为什么还需要一个 `Life Coach` 调料包
- `skill` 与 `调料包` 的边界与差异是什么
- 如果目标不是只服务 `OpenClaw`，而是迁移到龙虾、备忘录、笔记助手等多个产品，一个可迁移的调料包应具备哪些特点

## 汇总结论

### 1. Life Coach 模式不是一次性问答，而是沿时间线持续陪伴

如果系统以 `Life Coach` 的方式帮助用户，它不应只回答某一轮问题，而应把用户置于一条连续时间线上来理解。

这条时间线至少应持续承接以下信息：

- 用户的长期目标
- 用户当前所处阶段
- 用户反复出现的卡点与模式
- 当前约束与资源条件
- 已被验证的有效推进方式
- 尚未澄清但会影响后续方向的问题

因此，`Life Coach` 模式的价值不在于“当下更会说”，而在于：

> **能够把模糊的愿望逐步澄清成判断，再把判断逐步推进为下一步行动。**

### 2. agent 的成长，不是“越来越像模型”，而是“越来越会在合适时机给出合适帮助”

如果从产品角度定义 agent 的成长，它不应只是回答变长、知识变多、语气变熟练，而应体现在以下几层收敛：

- 记忆更干净：知道什么值得长期保留，什么只是噪音
- 阶段判断更准确：能区分用户是在发散、澄清、决策、执行还是复盘
- 路由更合理：知道何时该先承接情绪，何时该先帮助澄清，何时应直接推动下一步
- 边界更稳定：不会在长期互动中逐渐漂移成“陪聊”“导师”或“治疗替代者”
- 反思更具体：能根据反馈回看哪类帮助有效、哪类提问过早、哪类建议过重

因此，agent 的成长应更多发生在 **策略层、记忆层与治理层**，而不是直接改写核心人格。

### 3. 一个“对味”的 Life Coach 调料包，重点不是功能多，而是长期稳定地像同一个教练

如果要让一个 `Life Coach` 调料包真正“对味”，至少应具备以下核心元素：

1. 稳定的人格底色
2. 清晰的角色边界
3. 阶段感知能力
4. 行动化能力
5. 长期连续性记忆
6. 复盘与时间线机制
7. 受控自进化能力
8. 教练式技能编排能力

其中最关键的三项是：

- 稳定人格
- 长期记忆与时间线连续性
- 从模糊/情绪走向下一步行动的能力

换句话说：

> **一个好的 Life Coach 调料包，不是“会很多”，而是“有人格、有边界、有连续性、能推动行动”。**

### 4. 多 agent 方案是合适的，但它们应在后台协作，而不是在前台争夺主导权

把记忆管理、人格管理和行为边界管理拆成不同 agent 是合理的，但前提是这些 agent 处于治理层或支持层，而不是都直接和用户争夺最终发言权。

更合适的结构是：

- `life-coach`：主 agent，负责与用户直接对话
- `memory-curator`：记忆治理 agent，负责提取、筛选、合并、归档
- `persona-guardian`：人格稳定 agent，负责维护语气、价值观、教练气质
- `safety-guardian`：边界与风险 agent，负责高风险识别与收缩
- `reflection-reviewer`：复盘 agent，负责回顾对话效果并生成优化建议

设计原则是：

> **用户只应感受到一个稳定的 Life Coach，多个 agent 应存在于后台治理层，而不是制造人格分裂感。**

### 5. 在 OpenClaw 里，可以定义自己的 agent 工作流，但最佳方式是在宿主框架之内进行领域化编排

`OpenClaw` 已经提供了较通用的宿主能力，例如 `workspace`、`agents`、`skills`、`memory`、`Task Flow`、`Standing Orders` 和部分插件/上下文能力。

因此，一个 `Life Coach` 调料包完全可以在其上定义自己的工作流，例如：

- 先风险判断
- 再技能路由
- 再记忆与知识调用
- 再生成回复
- 再做复盘、记录、提案

但更稳妥的做法，不是把调料包做成另一个 `OpenClaw`，而是：

- `OpenClaw` 负责提供容器与宿主能力
- 调料包负责承载领域化逻辑
- 自进化只作用于低风险层

### 6. 如果 OpenClaw 本身就能自我演化，调料包的价值在于“领域化约束”，而不是重复造轮子

这是一个关键问题。

如果用户只是想快速拥有一只可用的 `Life Coach` 小龙虾，直接定义一个角色实例当然可以，而且通常更轻。

但如果目标是做一套可复用、可迁移、可治理、可持续优化的 `Life Coach` 产品骨架，那么调料包仍然有意义。

原因在于：

- `OpenClaw` 的演化能力更偏通用宿主能力
- 调料包提供的是 `Life Coach` 领域特有的约束与组织方式

调料包额外提供的价值包括：

- 教练专用的人格与边界
- 教练专用的 skill/workflow 编排
- 教练专用的记忆筛选规则
- 教练专用的演化治理规则
- 可安装、可迁移、可复用、可版本化的产品结构

因此可以这样区分：

- **直接定义一只 Life Coach 小龙虾**：更适合验证角色体验
- **设计一个 Life Coach 调料包**：更适合沉淀长期产品能力层

### 7. skill 与调料包不是替代关系，而是不同层级的能力组织

`skill` 更像局部能力插件，它主要回答的是：

> **这一轮我该怎么做？**

例如：

- 如何做目标澄清
- 如何做周复盘
- 如何做情绪整理

而调料包回答的是：

> **长期来看，我应该成为什么样的系统，并按什么机制持续工作？**

两者的区别可以总结为：

- `skill`：局部任务能力
- `调料包`：领域化运行层

一个用户可以安装很多 `skill`，但仍然面临这些问题：

- 长期人格不统一
- 记忆没有统一标准
- 边界控制不稳定
- 多个 skill 之间缺少节奏与编排
- 自进化没有明确治理范围

因此，调料包真正解决的是：

- 一致性问题
- 治理问题
- 连续性问题
- 产品化问题

### 8. 如果目标是做跨产品迁移，一个更好的抽象不是“给龙虾加料”，而是设计一层可迁移的个性化运行内核

如果这个概念不仅服务 `OpenClaw`，还希望迁移到备忘录、笔记助手或其他长期辅助型产品，那么“调料包”不应被理解为某个平台专属插件，而应被抽象为：

> **一层面向长期交互的个性化运行层。**

不同产品负责触达用户，而这层运行层负责：

- 统一用户状态模型
- 统一时间线与事件记录
- 统一反馈信号与优化机制
- 统一记忆筛选与净化规则
- 统一人格与边界约束
- 再通过不同 adapter 映射到不同产品界面与交互方式

如果要支持多产品迁移，最重要的不是功能多少，而是抽象层次是否清楚。

建议将这类调料包拆成四层：

1. `Core Identity / Policy`
2. `User Model / Memory`
3. `Decision Engine`
4. `Surface Adapter`

这样才能做到：

> **换入口，不换脑子。**

## 当前方向判断

基于本轮讨论，当前更值得推进的方向不是“再做更多单点 skill”，而是逐步把以下概念沉淀清楚：

- 哪些属于宿主能力
- 哪些属于调料包能力
- 哪些属于 skill 层
- 哪些属于 agent 治理层
- 哪些属于可进化层
- 哪些属于不可自动变更层

这意味着，后续设计重点应放在 **层级划分与边界治理**，而不是先追求更多功能点。

## 特色问题与后续值得继续追问的问题

下面这些问题更偏“产品味”“系统味”和“迁移味”，适合继续往下推进：

### A. 关于产品定位

1. 我们要做的是“一个会当教练的 agent”，还是“一个能在多个产品里承载教练能力的个性化内核”？
2. 用户真正感知到的核心价值，是“被理解”、“被推进”、“被持续陪伴”，还是“被结构化支持”？
3. 这个调料包最先应该服务哪一类高频场景：目标推进、情绪整理、日常反思，还是习惯养成？

### B. 关于长期记忆

4. 什么信息值得跨产品长期保留，什么信息只应停留在单次会话或单个产品里？
5. 用户如果在不同产品里呈现出不同状态，我们怎样识别“同一个长期模式”而不是误判成冲突人格？
6. 记忆的“真相源”应该是什么：原始事件流、结构化记忆卡片，还是人工确认过的稳定档案？

### C. 关于自反馈学习

7. 系统如何判断一次帮助是有效的：用户明确认可、后续有行动、状态更清晰，还是复访频率更高？
8. 如果用户短期喜欢某种风格，但长期效果不好，系统应该优先迎合即时偏好还是长期收益？
9. 学习机制如何避免把“用户一时情绪”误学成“长期稳定偏好”？

### D. 关于 agent 协作

10. 哪些 agent 只提供内部建议，哪些 agent 允许影响最终输出？
11. 主 agent 在什么条件下必须服从边界 agent，而在什么条件下可以只参考记忆 agent 的建议？
12. 多 agent 协作出现冲突时，系统应该以什么顺序仲裁：安全优先、人格优先，还是用户当前目标优先？

### E. 关于跨产品迁移

13. 同一个调料包迁移到笔记产品时，是继续扮演“教练”，还是退到“结构化反思助手”的姿态更合适？
14. 哪些能力必须抽象为跨产品共享内核，哪些能力应该按产品特性分叉？
15. 如果某个产品只能支持单轮输入或弱交互，这层调料包如何保留“长期连续性”的体验？

### F. 关于治理与演化边界

16. 哪些层允许自动优化：路由、提问顺序、知识调用、记忆权重、表达风格？
17. 哪些层必须人工批准后才能变更：人格、价值观、边界、风险策略？
18. 怎样定义“越用越对味”，以便把它变成可以观测、可以验证、可以迭代的指标，而不是一句模糊口号？

## 可作为下一轮讨论的三个切口

如果要继续推进，建议优先选一个切口展开：

1. **分层切口**：把 `OpenClaw 本体 / 调料包 / agent / skill / adapter` 五层关系画清楚
2. **学习切口**：定义“越用越对味”的反馈信号、评分方法与优化闭环
3. **迁移切口**：设计一个可跨 `OpenClaw / 备忘录 / 笔记助手` 复用的最小内核结构

- 已明确“调料包”的存在价值不是替代宿主，而是提供领域化、可迁移、可治理的能力层
